# reclip — план и открытые вопросы

## 🚧 Глобальный план реализации (MVP)

Каждый этап — самостоятельный кусок: собирается и проверяется отдельно.
Порядок важен (каждый следующий опирается на предыдущий).
Статусы: ⬜ не начат · 🟨 в работе · ✅ готов

- ✅ **Этап 0 — Каркас + проверка риска.** (2026-07-07)
  Создан `Cargo.toml` (пока минимальный: `arboard`+`anyhow`, полный набор 8.3
  развернём на Этапе 1) и `src/main.rs` — проба чтения буфера.
  **Результат — оба риска сняты:** (а) `arboard` читает буфер на GNOME/Wayland
  через X11-бэкенд/XWayland — план Б (`wl-clipboard-rs`/`xclip`) не нужен;
  (б) `cargo` тянет крейты через proxychains без доп. настройки прокси.
  Раскладку модулей (7.1) разворачиваем на Этапе 1.

- ✅ **Этап 1 — Модель + хранилище.** (2026-07-07)
  `model.rs` (`ClipItem`, enum `Content`, превью-строка), `storage.rs`
  (SQLite WAL, схема + `PRAGMA user_version`, `add_text` с фильтрами 1.4/1.5,
  дедуп-подъём через «удалить+вставить» → свежий id наверху, обрезка до 100).
  Каркас CLI на `clap` (под-команды-заглушки). **11 юнит-тестов зелёные.**
  Зависимости этапа: clap, rusqlite(bundled), anyhow, directories.

- ✅ **Этап 2 — Абстракция буфера.** (2026-07-07)
  `clipboard.rs`: trait `Clipboard` (get_text/set_text, 7.2), реализация
  `ArboardClipboard` (пустой буфер → `Ok(None)`, не ошибка), плюс `MockClipboard`
  для тестов логики без дисплея. Проверено: мок-roundtrip в `cargo test` и
  живой roundtrip через `arboard` (`cargo test -- --ignored`) — оба зелёные.

- ✅ **Этап 3 — Демон.** (2026-07-07)
  `daemon.rs` + под-команда `reclip daemon`: single-instance через `flock`
  (`fs2`) на `$XDG_RUNTIME_DIR/reclip.lock` (3.3); цикл `poll_once`+sleep 500 мс
  (3.1); чистая `poll_once` (буфер→last_seen→`add_text`) под юнит-тесты.
  Логи `log`+`env_logger`, уровень `info` по умолчанию (8.2). Разовые ошибки
  чтения буфера не роняют демон. 6 юнит-тестов (через `MockClipboard`) +
  живой smoke-тест: подтверждены single-instance, фильтр пробелов, дедуп-подъём.
  *Замечание: перед ручным запуском нужен `cargo build` — `cargo test` не
  обновляет бинарь `target/debug/reclip`.*

- ✅ **Этап 4 — CLI-просмотр (отладочный).** (2026-07-07)
  Под-команда `reclip list`: нумерованный список превью (новейшее сверху,
  ширина 80), «История пуста.» на пустой базе (8.4). Сквозной живой тест
  демон→база→list подтвердил дедуп-подъём, схлопывание переносов, обрезку с «…».

- ✅ **Этап 5 — Пикер (GUI).** (2026-07-07)
  `picker.rs` + под-команда `reclip show`: окно GTK4, список превью (4.5),
  навигация ↑/↓/Enter/Esc/цифры 1-9 (4.4), выбор кладёт в буфер и закрывает
  (4.2), закрытие по Esc/потере фокуса/выбору (4.3). Пикер только читает базу
  (5.1). Буфер пишем через GDK (родной клиент GTK — GNOME сохраняет содержимое
  после выхода); закрытие с задержкой ~120 мс, чтобы композитор принял владение.
  Ручной тест пройден: вставка (в т.ч. персистентность), цифры, Esc, потеря
  фокуса — всё работает.
  *Известные ограничения Wayland (не баг): окно не центрируется программно —
  позицию задаёт композитор (4.3). Правки по ходу: фокус ставим на первую
  строку через `GtkWindowExt::set_focus` после `present()`, иначе ↑/↓ не
  работали до первого нажатия.*

- ✅ **Этап 6 — Запуск и установка.** (2026-07-07)
  `dist/reclip.service` — systemd user-сервис (PartOf/WantedBy graphical-session,
  Restart=on-failure). `install.sh` — проверка зависимостей (cargo/cc/gtk4),
  `cargo build --release`, бинарь в `~/.local/bin`, сервис в `~/.config/systemd/
  user`, `enable`+`import-environment`+`restart`, инструкция по `Super+V` (4.1).
  Проверено вживую: сервис active+enabled, демон пишет буфер, `reclip list`
  показывает свежие копирования. Остаётся ручной шаг — назначить Super+V.

- ✅ **Этап 7 — Полировка MVP.** (2026-07-07)
  Сквозной ручной тест пройден на живой системе: «скопировал → Super+V → выбрал
  → вставил» работает. `README` переписан под готовый MVP (возможности,
  установка, использование, устройство, ограничения, удаление). Версия поднята
  до `0.1.0`. Итог: 18 юнит-тестов зелёные, сборка без предупреждений.

  **🎉 MVP готов.**

## 🖼 Фича: картинки в буфере (сверх MVP)

Решения — в `docs/09-images.md`. Этапы по той же схеме: собирается и
проверяется отдельно. Статусы: ⬜ не начат · 🟨 в работе · ✅ готов

- ✅ **Этап И0 — Проба риска.** (2026-07-19) Пробник `examples/image_probe.rs`,
  три проверки на живом GNOME/Wayland: (а) автоматический roundtrip
  запись→чтение — байт-в-байт; (б) чтение чужой картинки (скриншот 657×505) —
  размер точь-в-точь; (в) запись → вставка в реальное приложение — ок.
  **Риск 9.0 снят: `arboard` полностью тянет картинки, план Б не нужен.**
  Пробник оставлен до Этапа И5 (ручные проверки И2/И4), там удалим.
- ✅ **Этап И1 — Модель + хранилище.** (2026-07-19) `Content::Image {png,width,
  height}` + превью «🖼 Ш×В» + аксессор `ClipItem::image()`; миграция схемы v2
  (пересоздание таблицы с nullable `text` и колонками `image/width/height/hash`,
  перенос старых текстовых строк — 9.4); `add_image(rgba,w,h)`: хеш по сырому
  RGBA (дедуп 9.2), кодирование в PNG через крейт `image` (9.1), лимит 10 МБ и
  отбраковка битого RGBA (9.3); `list()` читает оба типа. **26 юнит-тестов
  зелёные** (в т.ч. живая миграция v1→v2 без потери записей). Зависимость
  этапа: `image` (png, чистый Rust). *Нюанс: `PngEncoder::write_image` при
  неверной длине буфера паникует — длину проверяем сами до вызова.*
- ✅ **Этап И2 — Абстракция буфера.** (2026-07-19) Тип `ClipImage {width,height,
  rgba}`; trait `Clipboard` расширен `get_image`/`set_image`; реализация на
  `arboard` (RGBA ↔ `ImageData`, пустой буфер/не-картинка → `Ok(None)`);
  `MockClipboard` хранит картинку, текст и картинка взаимно вытесняются (как в
  реальном буфере). **28 юнит-тестов** + живой `#[ignore]`-roundtrip картинки
  через arboard прошёл байт-в-байт (`cargo test -- --ignored`).
- ✅ **Этап И3 — Демон.** (2026-07-19) `poll_once` опрашивает картинку после
  текста (приоритет текста, 9.7); тип `LastSeen::{Text,Image}` различает
  «то же, что в прошлый тик» — текст целиком, картинку по хешу RGBA (общий
  `storage::rgba_hash`, чтобы «та же картинка» совпадала с дедупом хранилища),
  что не даёт переупаковывать PNG каждые 500 мс. **32 юнит-теста** через мок:
  добавление картинки, неизменная не пишется повторно, переключение
  текст↔картинка, эхо-подъём картинки. *Живую сквозную проверку демона
  (копирование картинки → история) делаем на Этапе И4 вместе с пикером.*
- ✅ **Этап И4 — Пикер.** (2026-07-19) Строка-картинка = миниатюра (высота 80,
  пропорции сохранены, `gdk::Texture::from_bytes` → `gtk::Picture`) + подпись
  «🖼 Ш×В» (9.5); выбор картинки кладёт её в буфер через `set_texture` (9.6),
  при сбое декодирования — откат на подпись/не роняем окно. Включена фича
  `gtk4/v4_6`. **Живой сквозной тест пройден** (пользователь): демон записал
  реальный скриншот, `Super+V` показал миниатюру+подпись, выбор → вставка
  картинки работает, текстовые записи не сломаны. Заодно вживую подтверждён
  Этап И3 (демон ловит картинки) и миграция v1→v2 на боевой базе без потерь.
- ✅ **Этап И5 — Полировка.** (2026-07-19) `reclip list` показывает картинки как
  «🖼 Ш×В» (через `preview`, проверено вживую); README обновлён (картинки в
  «Возможностях», убраны из «Ограничений», формат хранения PNG, ссылка на
  `docs/09`); версия поднята до **0.2.0**; временный пробник
  `examples/image_probe.rs` удалён. **32 юнит-теста зелёные**, релиз собирается
  без предупреждений (кроме proxychains-артефакта линкера в песочнице).

  **🎉 Картинки готовы (версия 0.2.0).**
