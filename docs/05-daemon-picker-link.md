# 05. Связка демон ↔ пикер

Решения по группе 5 из `TODO.md`.

## 5.1 Проблема «эха» — ✅ решается дедупом + единственным писателем
Пикер кладёт в буфер запись, которая **уже есть в истории** (он её оттуда и
взял). Демон, увидев это изменение буфера, применяет обычное правило из 1.3
(«есть в истории → поднять наверх»). То есть «эхо» само превращается в
правильное поведение: только что выбранное становится самым свежим.

Схема ролей:
- **Пикер НИКОГДА не пишет в базу.** Только читает историю (показать список)
  и кладёт выбранную запись в буфер.
- **Демон — единственный писатель.** Видит любое изменение буфера (в т.ч. от
  пикера) и применяет правило «есть → наверх, нет → добавить».

Плюсы: нет «эха» как проблемы, один писатель (нет гонок за запись), нет спец-кода.
Мелочь: подъём наверх происходит с задержкой до 500 мс (следующий поллинг) —
незаметно, т.к. в буфер запись легла мгновенно.

## 5.2 Канал общения — ✅ без сокета в MVP
Отдельный канал (сокет) в MVP **не нужен**. Общение идёт через:
- **общую базу SQLite** — пикер читает, демон пишет (WAL разрешает
  одновременные чтение/запись);
- **системный буфер** — пикер кладёт, демон замечает.

Локальный сокет добавим позже, только когда появится фича, которой он реально
нужен (например, пауза записи или мгновенное обновление открытого окна).
