# 02. Хранилище истории

Решения по группе 2 из `TODO.md`.

## 2.1 Где лежит файл — ✅ `~/.local/share/reclip/history.db`
По стандарту XDG пользовательские данные кладём в `XDG_DATA_HOME`
(по умолчанию `~/.local/share/`). Путь: **`~/.local/share/reclip/history.db`**.
(Настройки, когда появятся, пойдут отдельно в `~/.config/reclip/` — см. группу 6.)

## 2.2 Формат — ✅ SQLite (rusqlite, bundled)
Выбран **SQLite** через крейт `rusqlite` с флагом **`bundled`**.

Важно понимать: SQLite — не сервер БД, а **встраиваемая библиотека**. Флаг
`bundled` вкомпилирует её C-исходники прямо в наш бинарь, поэтому на выходе
остаётся **один исполняемый файл**, ничего доустанавливать пользователю не надо.
Вся «база» — один файл `history.db`.

Почему не JSON — SQLite даёт под наш продукт:
- безопасный одновременный доступ (см. 2.4) без ручных трюков;
- инкрементальную запись (меняется одна строка, а не переписывается весь файл);
- готовность к картинкам (blob) и к поиску/сортировке/закреплению (SQL) в будущем.

Честная цена: при сборке нужен C-компилятор (на Ubuntu обычно есть),
файл не читается «глазами» (нужен `sqlite3`), сборка чуть дольше.

## 2.3 Живучесть между перезагрузками — ✅ да
Демон при старте открывает `history.db` и продолжает накопленную историю.

## 2.4 Одновременный доступ — ✅ полагаемся на SQLite (WAL)
Используем режим **WAL** (`PRAGMA journal_mode=WAL`) — SQLite сам безопасно
разруливает одновременные чтение/запись. Ручные приёмы «temp+rename» не нужны.
Принцип «пишет в основном демон» сохраняем, но пикер при необходимости тоже
может писать безопасно (детали связки — группа 5).

## 2.5 Версионирование формата — ✅ `PRAGMA user_version`
Версию схемы храним внутри базы через встроенный механизм SQLite
`PRAGMA user_version`. При обновлении программы код сверяет версию и при
необходимости выполняет миграцию схемы.
