# 09. Картинки в буфере обмена (первая фича сверх MVP)

Решения по поддержке изображений. Это первый пункт backlog после MVP (см.
`TODO.md`). Затрагивает все пять модулей: `model`, `storage`, `clipboard`,
`daemon`, `picker`. Порядок реализации — этапы «И0…И5» в `TODO.md`.

## 9.0 Риск — умеет ли `arboard` картинки на GNOME/Wayland — ✅ проверяем первым делом
Для текста мы сняли этот риск на Этапе 0 (arboard читает буфер через
X11/XWayland). Для картинок это **отдельная неизвестность**: `arboard` даёт
`get_image`/`set_image` (сырой RGBA), но поведение на конкретном GNOME/Wayland
надо подтвердить живой пробой (Этап И0) — чтение скопированной картинки и
запись с последующей вставкой в реальное приложение. План Б при провале —
GDK / `wl-clipboard-rs`.

## 9.1 Что храним — ✅ PNG в BLOB
Из буфера картинка приходит как **сырой RGBA** (`arboard::ImageData`:
`width`, `height`, `bytes` = 4 байта на пиксель). Сырой RGBA огромен
(1920×1080 ≈ 8 МБ), поэтому перед сохранением **кодируем в PNG** (без потерь) и
кладём в SQLite как `BLOB`. При вставке — декодируем PNG обратно в RGBA. Кодек —
крейт `image` (чистый Rust, без системных зависимостей).

## 9.2 Дедуп — ✅ по хешу байтов картинки
Как текст дедупим по равенству строки, картинку — по **хешу** её содержимого
(хеш сырого RGBA до кодирования — стабилен, не зависит от параметров PNG).
Повтор поднимаем наверх тем же приёмом «удалить+вставить». Это же решает «эхо»
(5.1): пикер вернул картинку в буфер → демон увидел → поднял наверх, дубля нет.
Хеш берём из `std` (`DefaultHasher`), без лишних крейтов: для дедупа в пределах
100 записей коллизии практически невозможны.

## 9.3 Лимит размера — ✅ PNG крупнее 10 МБ не сохраняем
По аналогии с порогом 1 МБ для текста (1.5): если **закодированный PNG** больше
**10 МБ** — запись не сохраняем (буфер работает штатно). Порог позже вынесем в
конфиг вместе с текстовым.

## 9.4 Схема БД — ✅ миграция v2, картиночные колонки
Сейчас таблица `items` имеет `text TEXT NOT NULL` — картинке текста нет. Миграция
**v2** (через `PRAGMA user_version`, 2.5) пересоздаёт таблицу с колонками под оба
типа, перенося старые текстовые записи:

```
items(
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    kind       TEXT    NOT NULL,          -- 'text' | 'image'
    text       TEXT,                      -- заполнено для kind='text'
    image      BLOB,                      -- PNG-байты для kind='image'
    width      INTEGER,                   -- размеры картинки (для подписи)
    height     INTEGER,
    hash       TEXT,                      -- ключ дедупа для картинок
    created_at INTEGER NOT NULL
)
```

Порядок в списке по-прежнему `id DESC`, обрезка до 100 — как есть.

## 9.5 Показ в пикере — ✅ миниатюра + подпись
Строка с картинкой = **миниатюра** (высота ~80 px, вписанная по пропорции) плюс
подпись вида «🖼 1920×1080». Навигация ↑/↓ / Enter / Esc / цифры 1–9 — без
изменений (4.4). Миниатюру строим из PNG через `gdk::Texture` → `gtk::Picture`.

## 9.6 Вставка из пикера — ✅ через GDK-текстуру
Как и текст (docs/04, причина — GNOME сохраняет буфер после выхода окна),
картинку кладём в буфер **через GDK**: `clipboard.set_texture(&texture)`, где
текстуру строим из PNG-байтов записи. Закрытие с той же задержкой ~120 мс, чтобы
композитор принял владение.

## 9.7 Демон — ✅ опрашивает картинку рядом с текстом
Порядок в `poll_once`: сначала текст (`get_text`), если текста нет — картинка
(`get_image`). `last_seen` расширяем, чтобы различать «тот же текст» и «та же
картинка» (по хешу), — не переписывать одно и то же каждые 500 мс. Чтение
картинки на каждом тике тяжелее текста, но без событий об изменении буфера
альтернативы нет; для MVP-объёма приемлемо.

## 9.8 Зависимости — ✅ только `image`
Добавляем один крейт — `image` (PNG encode/decode), чистый Rust. GDK/`gdk-pixbuf`
уже приходят с `gtk4`. Хеш — из стандартной библиотеки.
